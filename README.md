# HHHGraph Land
This repo generates the files, distances and adjacency needed for GNN training for HHH signal vs. background discrimination.

## Configuration
`config/user_<name>.yaml` contains user specific settings, currently including the paths where your inputs/outputs should be stored, and what signal model you are running over.
`config/ml_default.yaml` contains the ML architecture setup, including the number/nature of GCN/MLP layers, dropout and learning rates, LL, number of epochs.
`config/ml_<label>.yaml` can be made and would contain the ML architecture setup, for any other particular architecture you want to save for future use, just give it a label.

## Dataset splitting
`write_files.py` splits the root ntuples into training, validation and test datasets, and outputs them as h5 files.
It currently assumes one type of background is present! 
Your user config must be given so it knows where the ntuples live and where to store the h5 files to.
```
python write_files.py -u config/user_<name>.yaml
```

## Distance calculation
Linking lengths requires calculating a specific type of pair-wise distance for a specified set of variables, for a given dataset.
These can be specified when running
```
python calc_distance.py -v <type of variables> -d <distance> -s -u config/user_<name>.yaml
```
where the variables are separated into types mass, angular, shape and combined.
The types of distances calculated are euclidean, cityblock and cosine, and the functions that calculate them can be found in `utils/distances.py`.
The `-s` option allows for sampling of a given dataset, as the training dataset is quite large, the calculation for pair-wise distances is computationally intensive.
The distances and the pair-wise event weights are saved in `.pt` files in your 'dist_path'.
The types of variables that can be specified (for HHH signals) are `mass`, `angular`, `shape`, `combined`, `mass_and_angular` and `mass_and_shape`.
The types of distances that can be specified are `euclidean`, `cityblock` and `cosine`.

## Linking lengths
Once the distances are calculated and saved, linking lengths can be generated between sig-sig and sig-bkg distances, or sig-sig and bkg-bkg distances. The linking length is decided from looking at the ROC curve between these distributions, and choosing the one with the largest AUC, and a point on the ROC curve that corresponds to a specific sig-sig efficiency (the fraction of links between signal events to other signal events). The ROC curves, and linking lengths at different sig-sig efficiencies can be generated by running:
```
python linking_length.py -v <type of variables> -d <distance>  -u config/user_<name>.yaml>
```
and the sig-sig vs. bkg-bkg, sig-sig vs. sig-bkg linking lengths are saved in json files in your 'll_path'.

## Adjacency matrix
The adjacency matrix that defines the edges of between events are generated using a specified linking length. This is calculated in `torch_adj_builder.py`, and it uses functions from the script `utils/adj_mat.py`.
```
python torch_adj_builder.py -c config/ml_<config file>.yaml -u config/user_<name>.yaml
```

The linking length used here is the same if generated from comparing sig-sig with bkg-bkg distances or from sig-sig with sig-bkg distances by specifying, just using the sig-sig efficiency.
The linking length that gives 0.6, 0.7, 0.8, or 0.9 sig-sig efficiencies can be chosen.

## Training
Training can be run for your choice of ML alg, by running: 
```
python torch_train.py -c config/ml_<config file>.yaml -u config/user_<name>.yaml
```
This will retrieve the kinematics from the input h5 events, and the adj matrix, and use these for training.
The config file should contain the name of the training, the type of kinematic variables, the type of distance metric, sig-sig efficiencies, and training hyperparameters.
For an out-of-the-box test use `ml_default.yaml`
The sig-sig efficiencies that are supported so far are `0.6`, `0.7`,` 0.8` and `0.9`. 
The model can be specified in the config file, including number of gcn and regular mlp layers and their sizes. Note: if there are no hidden gcn layers, model is equivalent to a regular dnn. 
The adjacency matrix is normalised by default in the GCNConv layer using the $`D^{-1/2} A D^{-1/2}`$ scheme used in the "Semi-supervised Classification with Graph Convolutional Networks" paper (https://arxiv.org/pdf/1609.02907.pdf).
